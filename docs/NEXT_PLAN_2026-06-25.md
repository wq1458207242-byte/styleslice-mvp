# StyleSlice 下一步开发计划

更新日期：2026-06-25

## 1. 下一阶段总目标

下一阶段的重点不是再堆更多按钮，而是把 StyleSlice 从“能跑通的 AI 画布”推进到“可稳定生产游戏 UI 资产”的工具。

核心目标：

- AI 接入更可靠。
- 生图更可控。
- 拆解更通用。
- 切片更可编辑。
- 项目数据更安全。
- 导出更接近 Unity 实际使用。

## 2. 优先级 P0：稳定 AI 基础设施

### 2.1 建立 Provider Profile 配置

当前 AI 配置已经支持 ModelScope 和 SiliconFlow，但还不够像“配置档案”。建议改成：

- 一个配置档案对应一个平台。
- 每个平台保存独立 API Key、Base URL、文字模型、视觉模型、生图模型、协议。
- 项目只记录当前使用的 profileId。
- 切换平台时不覆盖另一个平台的模型和 Key。

推荐内置 profile：

| 平台 | 适合能力 | 备注 |
| --- | --- | --- |
| SiliconFlow | 生图、OpenAI-compatible 文本/视觉 | 当前更容易跑通 |
| ModelScope | 国内模型、API-Inference、部分视觉/生图模型 | 额度和协议差异需注意 |
| Local Proxy / ComfyUI | 本地可控生图、后期生产化 | 最适合长期高频资产生产 |

### 2.2 拆分测试连接

不要再只有一个“测试连接”。改成四个明确测试：

- 测试文字模型。
- 测试视觉模型：上传一张小图，让模型回答图中主色或元素。
- 测试生图模型：生成一张 512 小图并下载为 dataURL。
- 测试本地代理：验证 `npm run api` 是否运行、图片转存是否正常。

每个测试要返回：

- 成功/失败。
- 实际请求的 URL。
- 使用的模型。
- 错误原因。
- 下一步建议。

### 2.3 明确模型能力

在配置面板增加能力标签：

- 文本：支持 / 不确定 / 不支持。
- 视觉：支持 / 不确定 / 不支持。
- 生图：支持 / 不确定 / 不支持。
- 异步任务：需要 / 不需要。

不要让用户用文本模型承担视觉任务，也不要让不支持 image endpoint 的模型承担生图任务。

## 3. 优先级 P0：生图策略重构

当前“一次生成整张组件板”的策略已经可以演示，但不够生产稳定。建议改为三层策略。

### 3.1 策略一：分组件生成

把组件板生成拆成：

1. 生成按钮。
2. 生成面板。
3. 生成进度条。
4. 生成徽章。
5. 生成头像框。
6. 生成对话框。

每个组件单独生成 2-4 个候选，然后统一排版成组件库。

优点：

- 每次任务更简单。
- 更容易控制提示词。
- 失败时只重试单个组件。
- 切片更准确。

### 3.2 策略二：生成后视觉质检

每次生图后用视觉模型检查：

- 是否是 UI 组件。
- 是否有多余角色/动物/人物。
- 是否符合参考风格。
- 是否边界清晰。
- 是否适合透明切片。
- 是否包含不可用文字或水印。

质检不通过则自动改提示词重试。

### 3.3 策略三：人工选择候选

生成结果不应直接覆盖。建议增加“候选池”：

- 每个组件生成多个候选。
- 用户选择最满意的。
- 可收藏、删除、重命名。
- 可发送到切片节点。

这会比“一键生成唯一答案”更符合真实设计生产。

## 4. 优先级 P0：拆解与切片编辑器

### 4.1 增加裁切框编辑

自动拆解永远会有误差，所以必须允许用户修正：

- 在图片上显示候选框。
- 拖拽移动框。
- 调整大小。
- 新建框。
- 删除框。
- 给框设置类型：按钮、面板、Icon、IP、字体、背景。
- 修改资产名称。

### 4.2 引入更强的本地图像处理

当前 Canvas 连通域检测是轻量方案。后续可考虑：

- `opencv.js`：边缘检测、轮廓检测、形态学处理更稳定。
- OCR：识别文字区域，辅助字体节点和文本剔除。
- 背景移除模型：处理 IP 或 Icon 的透明抠图。
- SAM/分割类模型：后期如果有本地或云端可用接口，可用于语义分割。

### 4.3 切片质量标准

切片节点应增加可视化质检项：

- 是否透明背景。
- 边界是否有残留背景。
- 是否有文字乱码。
- 是否尺寸过小。
- 是否适合九宫格。
- 推荐 pivot、border、pixels per unit。

## 5. 优先级 P1：节点系统增强

### 5.1 节点输入输出口语义化

现在节点主要是一个输入口和一个输出口。后续建议拆分：

- 参考图节点：图片输出、主图输出、风格证据输出。
- 风格包节点：风格输出、提示词输出、色板输出。
- 组件库节点：候选资产输出、分类结果输出。
- 切片节点：PNG 输出、metadata 输出。

这样连接会更清楚，也更适合复杂工作流。

### 5.2 节点运行策略

增加：

- 运行当前节点。
- 运行当前节点及上游。
- 运行当前节点及下游。
- 批量运行选中节点。
- 停止运行。
- 清空当前节点结果。

### 5.3 节点版本与结果缓存

每次 AI 生成都应保存：

- prompt。
- negative prompt。
- provider。
- model。
- seed 或 request id。
- 生成时间。
- 原图/结果图。
- 质量评分。

这样后续可以复现和比较，而不是每次生成后覆盖。

## 6. 优先级 P1：风格一致性优化

### 6.1 风格包拆成更精确的字段

当前风格包已有色彩、材质、形状、装饰。建议继续结构化：

- 主色、辅助色、强调色、背景色。
- 圆角半径倾向。
- 描边粗细。
- 阴影方向。
- 高光方式。
- 材质：软塑料、果冻、金属、羊皮纸、像素、木质等。
- 装饰元素：星星、齿轮、缎带、宝石、藤蔓等。
- 禁止元素：角色、真实照片、复杂背景、文字。

### 6.2 参考图证据可视化

在风格包节点中显示：

- 哪张图贡献了哪个色彩。
- 哪个区域体现了材质。
- 哪些组件被识别为按钮/导航/卡片。

这有助于用户判断 AI 是否真的理解参考图。

## 7. 优先级 P1：导出增强

Unity 导出建议增加：

- `sprites.json` 增加九宫格 border。
- 增加 Unity Editor 导入脚本。
- 支持按类型分文件夹：
  - Buttons
  - Panels
  - Icons
  - Characters
  - Backgrounds
- 支持导出命名规范预览。
- 支持单独导出选中资产。

建议下一轮按这个顺序做：

2. AI 配置改成 Provider Profile。
3. 测试连接拆成文字/视觉/生图/代理四项。
4. 生图策略改为分组件生成 + 候选池。

6. 切片节点增加九宫格编辑与透明背景质检。
7. Unity 导出增加 Editor 导入脚本。

## 10. 对产品方向的判断

这个产品最有价值的方向不是“生成一张漂亮 UI 图”，而是成为一个游戏 UI 资产生产台：

- 参考图进入。
- 风格被结构化。
- UI 元素被拆解。
- 组件被生成和筛选。
- 资产被切片和命名。
- Unity 可直接导入。

因此下一阶段应少做“泛泛的生图增强”，多做“资产生产链路的可控性”。AI 应该像一个高产但需要约束的美术助手，而不是一个黑箱按钮。
